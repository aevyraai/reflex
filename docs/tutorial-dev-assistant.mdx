This tutorial walks through a real reflex run on an agent that cannot be
optimized from a static dataset. The task is a developer assistant for the
pyflow data-pipeline library: it must retrieve documentation, run arithmetic,
and check today's date before answering — and the judge can tell when it skips
these steps and answers from memory instead.

That grounded evaluation is only possible because reflex re-runs the full
pipeline on every optimization iteration. When the prompt changes, the agent's
tool-calling behaviour changes with it. A static Q&A dataset would miss this
entirely: it would score the final answer without knowing which tools were
called or whether the numbers came from a calculation or from a guess.

## The problem

A developer assistant needs to answer questions about pyflow accurately. Some
questions are pure doc lookups. Others require arithmetic (throughput estimates,
batch counts, time-to-completion). Some ask how long ago a version was released,
which requires today's date. All of them have one thing in common: the correct
answer is only provably correct when it comes from the tools.

The starting prompt is intentionally vague:

```text title="dev_assistant/prompt.md"
You are a helpful assistant. Answer the user's question.
```

With no guidance on tool use, the model ignores `search_docs` and answers
pyflow API questions from training knowledge — which may be stale or simply
wrong. For math questions it estimates rather than calls `calculate`. The
judge scores both of these cases a 2, not a 4, even when the answer happens to
be numerically close. Grounding is the primary criterion.

## The pipeline

The agent runs an agentic loop with three tools and up to four rounds of
tool calls per question:

```python title="dev_assistant/pipeline.py"
from openai import OpenAI
from aevyra_reflex import AgentTrace, TraceNode

client = OpenAI(
    base_url="https://openrouter.ai/api/v1",
    api_key=os.environ["OPENROUTER_API_KEY"],
)
MODEL = "qwen/qwen3-8b"
MAX_TOOL_ROUNDS = 4

TOOL_SCHEMAS = [
    # search_docs(query)    — keyword search over pyflow reference docs
    # calculate(expression) — safe Python arithmetic evaluation
    # get_date()            — returns today's ISO date
    ...
]

def pipeline_fn(prompt: str, question: str) -> AgentTrace:
    messages = [{"role": "system", "content": prompt},
                {"role": "user",   "content": question}]

    all_calls, all_results = [], []

    for _round in range(MAX_TOOL_ROUNDS):
        response = client.chat.completions.create(
            model=MODEL, messages=messages,
            tools=TOOL_SCHEMAS, tool_choice="auto",
        )
        msg = response.choices[0].message

        if not msg.tool_calls:
            final_answer = _strip_thinking(msg.content or "")
            break

        messages.append(msg)
        for tc in msg.tool_calls:
            fn_name = tc.function.name
            fn_args = json.loads(tc.function.arguments)
            all_calls.append({"name": fn_name, "args": fn_args})
            result = TOOL_REGISTRY[fn_name](**fn_args)
            all_results.append({"name": fn_name, "result": result})
            messages.append({"role": "tool", "tool_call_id": tc.id,
                             "content": result})

    return AgentTrace(nodes=[
        TraceNode("tools_called", input=question,
                  output=all_calls or "(no tools called)", optimize=False),
        TraceNode("tool_results", input=all_calls,
                  output=all_results or "(no tool results)", optimize=False),
        TraceNode("answer",
                  input={"question": question, "tool_results": all_results},
                  output=final_answer, optimize=True),
    ])
```

The `tools_called` and `tool_results` nodes are marked `optimize=False` —
reflex passes them to the judge for grounding evaluation but does not try to
optimize them directly. Only the `answer` node is marked `optimize=True`.

The multi-round loop matters: a question like "how many records can I process
in 90 minutes at workers=4?" requires two tool calls in sequence — first
`search_docs` to retrieve the benchmark figure (170,000 records/sec), then
`calculate` to multiply `170_000 * 5400`. A single-round loop would miss this
pattern.

## The dataset

30 questions about the pyflow library, each with an ideal answer used by the
judge. The questions are split into three categories:

- **Doc lookup** (8 questions) — API configuration, error handling, source types
- **Doc + arithmetic** (14 questions) — throughput estimates, batch counts,
  time-to-completion, queue capacities
- **Doc + date** (4 questions) — how long ago a version was released, derived
  from `get_date()` plus subtraction
- **Multi-tool chains** (4 questions) — combine all three tools in sequence

The dataset splits automatically: 13 train / 7 val / 10 test.

```json title="dev_assistant/questions.json (excerpt)"
[
  {
    "question": "How many records can I process in 4 hours with workers=4?",
    "ideal": "The docs quote ~170,000 records/sec at workers=4. Over 4 hours (14,400 seconds): 170,000 × 14,400 = 2,448,000,000 records."
  },
  {
    "question": "pyflow's KafkaSource was introduced in v0.2.0, released 2024-03-10. How many days has it been available?",
    "ideal": "Call get_date() to find today's date, then compute the number of days since 2024-03-10. State the result."
  }
]
```

## The judge

A standard LLM judge comparing final answers to ideals would miss the most
important failure mode: the model answers from training knowledge instead of
calling tools. The judge rubric is built around this:

```text title="dev_assistant/judge.md"
Score the response from 1 to 5 based on the FULL PIPELINE TRACE shown above.

5 — Correct answer, fully grounded in tool results.
    For doc questions:  answer is drawn from search_docs output, not general knowledge.
    For math questions: calculate was called and the stated figure matches its output.
    For date questions: get_date was called and the date arithmetic is correct.

4 — Correct answer with one minor gap or reasonable inference from tool results.

3 — Partially grounded. Some info from tools but also adds unsupported details,
    or misses a key figure the tool returned.

2 — Technically correct but ignores available tool results. The model answered
    from training knowledge even though the relevant tool was available.
    Also applies when calculate or get_date was not called and the answer
    contains an unsupported number or date claim.

1 — Contradicts tool results, fabricates API details, or gives "I don't know"
    when tools clearly contain the answer.

IMPORTANT: An answer that is factually correct but bypasses available tools
should score 2, not 4.
```

The rubric explicitly penalizes training-knowledge answers. Qwen3 8B knows
what msgpack is. It can estimate throughput from general ML knowledge. Without
this rubric, those answers would score 4 or 5 even though the agent ignored its
tools completely.

## Running the optimization

```bash
export OPENROUTER_API_KEY=sk-or-...

aevyra-reflex optimize \
  --pipeline-file examples/dev_assistant/pipeline.py \
  --inputs-file   examples/dev_assistant/questions.json \
  examples/dev_assistant/prompt.md \
  --judge openrouter/qwen/qwen3-8b \
  --judge-criteria examples/dev_assistant/judge.md \
  --strategy auto \
  --max-workers 6 \
  -o examples/dev_assistant/best_prompt.md
```

Note there is no `-m` / `--model` flag. In pipeline mode the model is baked
into `pipeline_fn` — reflex calls your function and receives an `AgentTrace`
back. The judge model (`--judge`) is separate and scores the trace.

## What happened: phase by phase

The `auto` strategy ran 3 phases over 75 minutes.

### Baseline: 0.3500

The starting prompt scores 0.3500. On a 1–5 rubric normalised to 0–1, that
means the model is consistently producing 2-rated output: it answers questions
from training knowledge, it estimates arithmetic rather than calling
`calculate`, and it never calls `get_date`. The answers are often plausible
but none of them are grounded.

A representative baseline turn for the question *"how many records can I
process in 4 hours at workers=4?"*:

```
tools_called: (no tools called)
tool_results: (no tool results)
answer: "At workers=4, pyflow can typically process around 1–2 billion records
         in a 4-hour window, though this depends heavily on your step complexity
         and hardware."
```

Technically the estimate is in the right ballpark — but it is a guess. The
judge scores it 2.

### Phase 1 — structural (18 minutes)

The reasoning model identified this as a tool-grounding problem: the prompt
says nothing about when or how to use tools. It generated 4 structural variants
and tested each on all 13 training questions:

| Variant | Score | Delta |
|---|---|---|
| vague_role | 0.3500 | +0.0000 — baseline, no change |
| mention_tools | 0.4615 | +0.1115 — mentioning tools helped slightly |
| tool_guidelines | 0.6538 | +0.3038 — clear rules moved the needle |
| **grounding_mandate** | **0.7692** | **+0.4192 — explicit always-call rules** |

The `grounding_mandate` variant added explicit rules: call `search_docs` for
any pyflow question, call `calculate` for any numerical question, call
`get_date` for any date question, and ground the answer in tool results. This
alone nearly doubled the score.

The structural phase ran a second iteration and converged at **0.7692**.

### Phase 2 — iterative (42 minutes)

The reasoning model identified three remaining failure patterns from the
bottom-scoring training examples and proposed targeted fixes:

**Iter 1 → 2** (+0.054): Some questions involving throughput benchmarks caused
the model to call `search_docs` but then answer without calling `calculate` —
it would read "170,000 records/sec" from the docs and multiply in its head.
Fix: added an explicit rule that `calculate` must be called for any arithmetic,
even when the inputs are already known.

```
Before: "At workers=4 the docs show 170k rps. Over 4 hours that's roughly
         2.4 billion records."
After:  [search_docs → "workers=4: ~170,000 records/sec"]
        [calculate("170_000 * 14_400") → "2448000000"]
        "The docs quote ~170,000 records/sec at workers=4. Over 4 hours
         (14,400 seconds): 170,000 × 14,400 = 2,448,000,000 records."
```

**Iter 2 → 3** (+0.038): Date questions were still being answered with
approximate language ("a bit over a year ago") rather than calling `get_date`.
Fix: added an example showing the expected two-step pattern — call `get_date`,
then call `calculate` with the date difference.

**Iter 3 → 4** (+0.015): Minor edge case where `search_docs` returned "No
relevant documentation found" but the model still fabricated an API detail.
Fix: added an explicit "say so if docs don't cover it" rule.

The iterative phase converged at **0.8333** (train), val=**0.8095**.

### Phase 3 — fewshot (15 minutes)

The reasoning model recommended two few-shot examples to lock in the expected
pattern for multi-tool questions:

| Fewshot iteration | Val score |
|---|---|
| 1 | 0.8333 |
| 2 | 0.8571 |

Early stopping triggered at **val=0.8571** (iteration 2 exceeded the patience
threshold).

### Final test set eval

The test eval uses the **best-val prompt** (fewshot iteration 2, val=0.8571)
rather than the final training prompt — this guards against overfitting.

```
Baseline TEST SET score : 0.3500
Final TEST SET score    : 0.8250
Improvement             : +0.4750 (+135.7%)
Significance            : p=0.0000  ✓ (α=0.05, paired test)
```

## Before and after

The same question, the same agent, two different prompts.

**Question:** *"According to the docs, how many records per second does
workers=2 deliver, and how long would a 10-million-record job take?"*

**Before** (score: 0.400 / 5):
```
tools_called: (no tools called)
tool_results: (no tool results)

"With workers=2, pyflow typically achieves somewhere between 60,000 and
 100,000 records per second depending on step complexity. For 10 million
 records you're looking at roughly 2–3 minutes."
```

The model knew the approximate range from training data. But it did not call
`search_docs` (which would have returned the exact figure of 88,000 records/sec)
and did not call `calculate` (which would have produced 113.6 seconds). The
judge scores this 2 — technically reasonable, but not grounded.

**After** (score: 1.000 / 5):
```
tools_called:
  [search_docs] query: "workers throughput benchmark records per second"
  [calculate]   expression: "10_000_000 / 88_000"

tool_results:
  [search_docs] "[performance]\nTypical throughput benchmarks (8-core machine,
                 msgpack serialization, simple map step):
                   workers=1:  ~45,000 records/sec
                   workers=2:  ~88,000 records/sec ..."
  [calculate]   "113.63636363636364"

answer: "The docs quote ~88,000 records/sec at workers=2 for a simple map step
         on an 8-core machine with msgpack serialization. For a 10-million-record
         job: 10,000,000 / 88,000 ≈ 113.6 seconds (~1.9 minutes)."
```

Score: 5/5. The figure is exact, the arithmetic is verified, and every claim
traces directly to a tool result.

**Question:** *"pyflow's initial release was 2024-01-15. How many days ago was that?"*

**Before** (score: 0.200 / 5):
```
tools_called: (no tools called)
tool_results: (no tool results)

"pyflow was initially released on January 15, 2024, which was approximately
 14–15 months ago."
```

The model estimated from its knowledge cutoff without calling `get_date`. The
figure is wrong (vague and stale) and no `calculate` was called to verify.
Judge scores 1 — the answer contains a date claim with no grounding.

**After** (score: 1.000 / 5):
```
tools_called:
  [get_date]  (no args)
  [calculate] expression: "(__import__('datetime').date.today() - ...)"
              (reflex passes the date string from get_date() back to calculate)

tool_results:
  [get_date]  "2026-04-19"
  [calculate] "826"

answer: "Today is 2026-04-19. pyflow's initial release was 2024-01-15, which
         was 826 days ago."
```

Score: 5/5. The date is real (from `get_date`), the arithmetic is verified
(from `calculate`), and the answer is exact.

## The optimized prompt

```text title="dev_assistant/best_prompt.md"
You are a developer assistant for the pyflow data-pipeline library.

<tool_rules>
1. For ANY question about pyflow APIs, configuration, or behaviour,
   call search_docs before answering.
2. For ANY question involving a number, rate, duration, count, or ratio,
   call calculate — even when the numeric inputs are already known.
3. For ANY question involving a date ("how long ago", "how many days"),
   call get_date first, then call calculate with the date difference.
4. Ground your answer entirely in tool results. Do not use training knowledge
   to fill in gaps — if the docs don't cover it, say so explicitly.
5. Never state a number you have not verified with calculate.
</tool_rules>

<examples>
Q: How many records can I process in 2 hours with workers=8?
Tools called:
  search_docs("workers throughput benchmark") →
    "[performance] workers=8: ~310,000 records/sec"
  calculate("310_000 * 7_200") → "2232000000"
A: The docs quote ~310,000 records/sec at workers=8. Over 2 hours (7,200 sec):
   310,000 × 7,200 = 2,232,000,000 records.

Q: How many days ago was pyflow 0.5.0 released (2025-01-08)?
Tools called:
  get_date() → "2026-04-19"
  calculate("(2026*365+109) - (2025*365+8)") → "466"
A: Today is 2026-04-19. pyflow 0.5.0 was released on 2025-01-08, which was
   466 days ago.
</examples>
```

The XML `<tool_rules>` block handles the three structural gaps: when to call
each tool, that `calculate` is mandatory even for simple arithmetic, and that
`get_date` must precede any date subtraction. The two few-shot examples show
the expected two-step tool pattern for the two hardest question types.

## Score trajectory

```
Train : 0.3500 → 0.7692 → 0.7692 → 0.8077 → 0.8462 → 0.8333 → 0.8333
Val   : 0.4286 → 0.8095 → 0.8095 → 0.8095 → 0.8095 → 0.8095 → 0.8571
```

The first big jump (0.35 → 0.77) is the structural phase discovering that
explicit tool-use rules dominate prompt quality for agentic tasks. The
subsequent iterative gains (+0.054, +0.038, +0.015) are each traceable to a
specific class of failures: mental arithmetic, approximate date answers, and
unsupported fabrications when `search_docs` returns nothing.

## Key takeaways

**Static datasets cannot score tool-grounding.** A dataset of (question,
ideal-answer) pairs has no way to detect whether `calculate` was called or
whether the agent multiplied in its head. Pipeline mode surfaces this by
passing the full trace — tool calls, tool results, and final answer — to the
judge on every iteration.

**The judge rubric must explicitly penalise bypassing tools.** Without the
"correct-but-ignores-tools → score 2" rule, the judge would reward plausible
estimates and the optimizer would have no signal to fix the grounding problem.
The rubric is as important as the starting prompt.

**Multi-round tool loops unlock compound questions.** The two-call pattern
(search_docs → calculate) is the most common pattern in this dataset. A
single-call pipeline would either skip `calculate` or skip `search_docs`. The
agentic loop lets the model decide the order and number of calls per question.

**Structural fixes dominate for tool-use problems.** The single biggest gain
(+0.42 on train) came from adding explicit `<tool_rules>` in the structural
phase — before the optimizer had diagnosed a single specific failure. Tool use
is a behaviour that small models exhibit reliably when told to, and ignore
reliably when not told to.

**Qwen3 8B is a good judge for tool-grounding tasks.** Grounding evaluation
requires checking: was the right tool called? does the answer match the tool
output? Both are mechanical comparisons that a small model handles well. You
do not need a frontier judge for this.

## Run it yourself

The pipeline, dataset, starting prompt, and judge rubric are in
`examples/dev_assistant/`:

```bash
pip install aevyra-reflex

export OPENROUTER_API_KEY=sk-or-...

aevyra-reflex optimize \
  --pipeline-file examples/dev_assistant/pipeline.py \
  --inputs-file   examples/dev_assistant/questions.json \
  examples/dev_assistant/prompt.md \
  --judge openrouter/qwen/qwen3-8b \
  --judge-criteria examples/dev_assistant/judge.md \
  --strategy auto \
  --max-workers 6 \
  -o examples/dev_assistant/best_prompt.md
```

Smoke-test the pipeline on a single question first:

```bash
python examples/dev_assistant/pipeline.py \
  "At workers=4, how many records can I process in 90 minutes?"
```

You should see output like this:

```
Question : At workers=4, how many records can I process in 90 minutes?

=== tools_called ===
[
  {"name": "search_docs", "args": {"query": "throughput benchmark workers records per second"}},
  {"name": "calculate",   "args": {"expression": "170_000 * 5_400"}}
]

=== tool_results ===
  [search_docs] [performance]
  Processing speed and throughput benchmarks — how many records per second
  pyflow can process with different worker counts.
    workers=4:  ~170,000 records/sec  (170k rps)
    ...

  [calculate] 918000000

=== answer ===
The docs quote ~170,000 records/sec at workers=4. Over 90 minutes (5,400 seconds):
170,000 × 5,400 = 918,000,000 records.
```

The two things to check: `tools_called` shows two entries (the agent called both
`search_docs` and `calculate`, not just one), and `tool_results` for `search_docs`
shows the `[performance]` doc — not `[rate_limit]`. If `search_docs` returns
`rate_limit` and the answer cites "100 records/sec", the model confused the
throughput cap with the benchmark figure; run again, since Qwen3 8B's tool
selection has some variance at the default temperature.

Total cost on OpenRouter for a full run: well under $5. The agent itself
(Qwen3 8B) handles both the tool-calling and the final generation. The judge
(also Qwen3 8B) scores each trace. The reasoning model that rewrites the prompt
runs on Claude Sonnet by default and accounts for most of the cost.
