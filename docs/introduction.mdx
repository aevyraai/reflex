aevyra-reflex is a lightweight prompt optimizer. Give it a dataset, a system
prompt, and a target score — it diagnoses why your model is underperforming,
reasons about the failure patterns, and rewrites the prompt until scores match
your target.

**No YAML. No framework dependencies. No separate server to deploy.** Just
`pip install aevyra-reflex` and a single command.

```bash
pip install aevyra-reflex
aevyra-reflex optimize dataset.jsonl prompt.md -m local/llama3.1 -o best_prompt.md
```

All evaluation runs through [aevyra-verdict](https://github.com/aevyraai/verdict).

## Dashboard

Every run is immediately explorable in the built-in dashboard:

```bash
aevyra-reflex dashboard
```

No separate server, no build step — opens `http://localhost:8128` with score
trajectory charts, prompt diffs, reasoning analysis, and token usage. Branch
from any iteration to continue with a different strategy.

## Why reflex

**Lightweight.** No heavy framework dependencies. Just Python, standard
library, and `numpy` for PDO math. Installs in seconds and has no opinion
about the rest of your stack.

**Zero config.** No YAML files, no project setup, no adapter classes to write.
Point it at a JSONL file and a prompt file and it runs.

**Fully local.** Ollama is a first-class citizen. Use a local reasoning model
so nothing leaves your machine:

```bash
aevyra-reflex optimize dataset.jsonl prompt.md \
  -m local/llama3.1 \
  --reasoning-model ollama/qwen3:8b
```

**Agentic, not scripted.** Each iteration the reasoning model explains *why* it
made a change — you learn from the run, not just get an output. The causal
rewrite log tracks what helped, what had no effect, and what hurt, so the model
avoids repeating dead ends.

## What it does

Given a dataset and a starting prompt, aevyra-reflex:

1. Sets a target score — from a verdict benchmark, a live model comparison, or an explicit threshold
2. Runs a baseline eval with verdict to measure the starting score
3. Analyzes failure patterns and rewrites the prompt to fix them
4. Re-evaluates after each rewrite, iterating until the score meets the target
5. Returns a before/after comparison with the optimized prompt

## When to use it

- You ran verdict and model A beats model B — you want to close the gap through prompt engineering, not by switching models
- A model scores poorly on your eval — you want a better prompt, not a bigger model
- You're iterating on a system prompt and want to automate the feedback loop
- You want to understand *why* a prompt works (the analysis teaches prompt engineering)

## How it fits together

```
                ┌──────────────┐
                │   Dataset    │
                │  (JSONL)     │
                └──────┬───────┘
                       │
                       ▼
  ┌─────────────────────────────────────┐
  │         aevyra-reflex               │
  │                                     │
  │  ┌───────────┐   ┌──────────────┐  │
  │  │  Strategy  │──▶│  Reasoning   │  │
  │  │  (auto)    │   │  model (LLM) │  │
  │  └─────┬─────┘   └──────────────┘  │
  │        │                            │
  │        ▼                            │
  │  ┌───────────┐                      │
  │  │  verdict   │  ← evaluates each   │
  │  │  (evals)   │    prompt revision   │
  │  └───────────┘                      │
  └─────────────────────────────────────┘
                       │
                       ▼
              ┌────────────────┐
              │ Optimized      │
              │ system prompt  │
              └────────────────┘
```
